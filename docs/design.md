# Agent Meeting Room 设计文档

## 1. 产品定位

这是一个类似即时聊天窗口的协作系统，但它不是普通聊天工具，而是面向任务完成的“多角色会议室”。

每个会议室里可以同时存在：

- 真人角色
- AI Agent 角色

这些角色围绕同一个目标进行讨论、分工、执行、测试、评审和收口。系统重点不是“单次问答”，而是“持续推进直到任务完成或主持人停止讨论”。

本版本选择 Python Web UI，而不是桌面 UI，原因很明确：

- 适合 Linux 服务器部署
- 适合远程访问测试
- 更容易扩展成多会议室体系
- 更容易做复杂配置页面

## 2. 核心目标

1. 支持创建多个会议室，每个会议室都能长期保存聊天记录、任务、评审和记忆。
2. 支持真人和 AI 混合协作，其中主持人默认为真人。
3. 角色配置必须可视化，并且保存为 JSON 文件。
4. 每个角色可以绑定模型、提示词和多个工具。
5. 只要有人发消息，相关 Agent 就会开始思考，但只有与自己有关时才回复。
6. 讨论可以被主持人暂停、恢复或停止。
7. Agent 可以在自己的工作目录中生成架构文档、代码草案、测试结果等产物，并把结果贴回会话框。

## 3. 关键场景

### 3.1 新建会议室

用户在左侧区域创建一个新的会议室，填写会议室名称和目标。系统会自动为这个会议室建立：

- 独立的角色 JSON 配置文件
- 独立的工作目录
- 独立的消息、任务、记忆、评审记录

### 3.2 进入某个会议室

用户可以在会议室列表中切换当前进入的会议室。每个会议室互不干扰，历史信息长期保存。

### 3.3 配置角色

角色配置放在单独页面中，避免在主会话页塞太多表单。配置项包括：

- 角色名称
- 真人 / AI 类型
- 描述
- 绑定模型
- 多选工具
- 大文本提示词
- 启用状态

### 3.4 讨论推进

主持人或其他真人发出一条消息后：

1. 系统先根据消息内容和角色职责选出最适合起跑的 AI Agent。
2. 当前 Agent 执行完后，必须明确推荐下一位应该接力的 Agent。
3. 下一位 Agent 继续判断是否与自己相关，相关时再回复。
4. 如果链路已经收敛，则交给评审 Agent 或直接交还主持人。
5. 当所有 Agent 都认为没有明显问题时，系统会提示主持人决定继续还是结束。
6. 如果主持人暂停或停止，则自动接力立即终止。

### 3.5 执行型 Agent 场景

例如主持人说“请写一段程序”：

- 架构师 Agent 会给出结构方案，必要时在工作目录写入架构草案。
- 开发 Agent 会根据架构建议生成脚本草案，并决定交给测试还是继续补实现。
- 测试 Agent 在发现已有脚本后再开始测试，而不是前期无关乱发言。
- 评审 Agent 在确认链路闭环后，不再继续自循环，而是把决定权交回主持人。

### 3.6 暂停与停止

主持人可以通过按钮或消息执行：

- 暂停讨论
- 恢复讨论
- 停止讨论

暂停时不继续自动讨论。停止后会议室不再自动推进，除非重新恢复为活动状态。

## 4. 功能模块

### 4.1 多会议室模块

- 新建会议室
- 列出所有会议室
- 切换进入指定会议室
- 每个会议室拥有独立状态

### 4.2 角色配置模块

- 角色配置文件保存为 JSON
- 每个会议室独立拥有一个 roles.json
- 独立角色编辑页面
- 提示词输入框更大，便于配置复杂角色行为

### 4.3 工具模块

当前实现的工具包括：

- 任务拆解
- 架构草案
- 代码草案
- 产物检查
- 测试执行
- 评审总结
- 相关性判断
- 执行命令
- 切换目录
- 目录浏览
- 文本搜索
- 读取文件
- 写入文件
- 环境探针
- SSH 执行占位
- 知识库列表与知识库阅读
- 工作区快照
- 验收检查
- 风险矩阵
- 依赖扫描
- 文档生成
- 测试规划
- 缺陷记录
- 日志摘要
- 记忆沉淀

每个角色可以通过多选按钮持有多个工具。工具不是简单标签，而是会影响角色行为，并在某些情况下真正执行轻量操作。

### 4.4 会话模块

- 大尺寸会话区域，显示更多消息
- 区分系统消息、普通消息、评审消息
- 支持自动触发 Agent 讨论
- 支持手动触发 Agent 思考

### 4.5 任务模块

- 创建任务
- 指定负责人
- 设置验收标准
- 跟踪任务状态
- 通过任务驱动会议阶段变化

### 4.6 长期记忆模块

- 每个会议室独立保存长期记忆
- 用户可以手工写入记忆
- 评审结果会自动沉淀为记忆
- Agent 回复时会检索相关记忆

### 4.7 模型配置模块

- 统一管理 LLM 配置
- 可视化编辑 provider、model、base_url、api_key、temperature、max_tokens、thinking 开关
- 角色配置页可直接绑定模型

## 5. 架构设计

### 5.1 领域模型层

文件：models.py

职责：

- 定义会议室、角色、消息、任务、评审、模型、工具等数据结构

### 5.2 持久化层

文件：persistence.py

职责：

- 用 SQLite 保存会议室、消息、任务、记忆、评审和模型配置

### 5.3 角色配置层

文件：role_store.py

职责：

- 用 JSON 文件保存每个会议室的角色配置
- 角色配置是会议室级别的，而不是全局共享

### 5.4 工具执行层

文件：tooling.py

职责：

- 根据角色持有的工具执行轻量动作
- 在会议室工作目录中生成架构草案、脚本草案或测试结果

### 5.5 Agent 层

文件：agents.py

职责：

- 判断“当前消息是否与自己相关”
- 在相关时生成回复
- 调用工具
- 在启用 LLM 时调用模型，在关闭 LLM 时回退到本地规则策略

### 5.6 服务层

文件：services.py

职责：

- 多会议室管理
- 讨论线程编排
- 阶段推进
- 主持人口令处理
- 角色配置保存
- 工作目录管理

### 5.7 Web 层

文件：webapp.py、templates、static/style.css

职责：

- 提供多会议室界面
- 提供角色单独编辑页
- 提供模型配置页
- 提供大尺寸会话窗口

## 7. 讨论机制设计

### 7.1 基本规则

1. 真人发言后，系统会选出最合适的首位 AI Agent。
2. 每个 Agent 回复后都必须给出下一位推荐对象。
3. 系统按 handoff 顺序串行推进，而不是所有 Agent 同时广播回复。
4. 若评审认为当前已闭环，则自动提示主持人决定继续还是结束。
5. 若主持人暂停或停止，则自动讨论终止。

### 7.2 为什么要做相关性判断

如果所有 Agent 不分场景强行回复，讨论会迅速变成噪音。

例如：

- 在还没有代码产物时，测试 Agent 应该少说话
- 在已经有测试结果时，评审 Agent 才有必要发言
- 在刚开始需求讨论时，架构师和产品专家更应该先发言
- 在测试失败后，应该优先回到开发 Agent，而不是继续让其他角色插话

因此相关性判断是必要机制，而不是可选优化。

### 7.3 当前默认角色

- 主持人
- 架构师 Agent
- 开发 Agent
- 测试 Agent
- 产品专家 Agent
- 评审 Agent
- 安全审计 Agent
- 运维 Agent
- 文档 Agent
- 集成 Agent
- 数据分析 Agent

## 8. 工具机制设计

工具的目标不是替代 Agent 思考，而是让某些角色具备“执行能力”。

例如：

- 架构师可生成 architecture.md
- 开发 Agent 可生成 generated_script.py
- 测试 Agent 可尝试运行已有脚本并回传结果
- 评审 Agent 可根据结果给出评审结论
- 运维 Agent 可读取环境与日志信息
- 文档 Agent 可生成交付说明
- 安全审计 Agent 可补充风险和依赖扫描结果

这使会议室不仅能讨论，还能逐步形成产物。

## 9. 已考虑到但你未明确提出的点

为了让这个系统后续更可用，我额外加入了这些设计考虑：

1. 每个会议室单独工作目录，避免不同任务相互污染。
2. 角色编辑页独立出来，避免主页面配置过度拥挤。
3. 支持通过环境变量关闭真实 LLM，以便本地和远端稳定测试。
4. 角色工具和提示词都放在配置层，便于后续扩展新角色。
5. 讨论状态单独建模为 active / paused / stopped，避免只靠消息文本推断。

## 10. 风险与不足

1. 当前工具执行仍然是轻量型，距离完整自动化工程执行还差一步。
2. 相关性判断目前主要依赖规则和角色提示词，不是完整语义路由系统。
3. 会议室虽已支持多房间，但尚未加入用户权限和登录机制。
4. 目前仍是同步 Web 请求驱动的串行接力，不是 WebSocket 实时流式通信。
5. 长期记忆目前是关键词检索，不是向量检索。

## 11. 自我反思

### 11.1 逻辑是否自洽

是自洽的。

原因在于：

- 多会议室和长久记录已经闭环
- 角色 JSON 配置和工具多选已经落到配置层
- 会话、任务、评审、记忆彼此打通
- 主持人的暂停和停止控制已经纳入流程
- 执行型 Agent 已经能够生成产物并把结果贴回会话
- 串行 handoff 机制已经把“上一位决定下一位”和“最终交还主持人”落到了流程里

### 11.2 是否已经完美

还没有完美。

当前版本更像一个“结构正确、可运行、可继续扩展”的第一阶段产品，而不是最终成熟系统。

### 11.3 下一阶段最值得做的事情

1. 把轻量工具执行扩展成真正可配置的执行链。
2. 增加认证、权限和多人同时在线能力。
3. 为长期记忆加入向量检索。
4. 把讨论推进改造成实时流式刷新。

### 11.4 结论

这版设计已经明显更接近你的原始想法：

- 有多个会议室
- 有长期聊天记录
- 角色配置独立、细致、可视化
- 每个角色可以拥有多个工具
- 会自动讨论，但不会无关乱说
- 可以暂停、恢复、停止
- 能形成实际产物并回传到会话

从验证产品思想的角度看，这已经是一个合理且扎实的版本。
